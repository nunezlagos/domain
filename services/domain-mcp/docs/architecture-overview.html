1. El cuadro grande

Domain tiene dos modos de operación según quién maneja el LLM:

  • Modo MCP (lo que usás todo el día): el LLM vive en el agente externo (Claude/opencode). Domain expone tools domain_* por stdio JSON-RPC. El agente decide qué tool llamar y cuándo guardar memoria.
  • Modo runtime propio: agents/flows/crons del catálogo de domain ejecutan en server-side; el LLM se llama vía proveedores (Anthropic/OpenAI/Google/Ollama) y domain orquesta el loop tool-use.

Los dos modos comparten Postgres, RLS, policies, memoria y skills. El usuario nunca elige uno explícitamente — el flujo cae naturalmente según qué tool llama.

2. Bootstrap del agente — antes del primer saludo

Cuando abrís opencode, domain ya está peleando para ser el sistema de memoria elegido. Esto ocurre antes de cualquier interacción con el usuario:

Doble capa de protocolo: el handshake MCP (capa débil, el cliente decide cuánto inyecta) trae el protocolo completo; el archivo de instructions del agente (capa fuerte, va al system prompt) trae un STUB que ordena cargar la policy viva. Editar la policy agent-protocol en BD cambia el comportamiento de TODOS los agentes sin tocar archivos.

3. Triage del mensaje — el agente decide qué hacer

El "triage" no vive en domain: es el LLM del agente externo aplicando las instructions. Domain solo provee el menú de tools y el contexto. Para un mensaje típico:

Mensaje del usuarioTool elegido por el LLMPor qué
"hola, ¿qué hicimos ayer?"domain_mem_context + domain_mem_searchprotocolo: al inicio de sesión recuperar contexto

"voy a tocar el schema de users"domain_policy_list + domain_policy_get(slug='db')protocolo: rules antes de tocar código
"corré el flow de deploy"domain_flow_runtool de catálogo

Token budget

Pre-flight valida que token_budget alcanza vs context_size del registry. Soft/hard limits clampean en runtime.

Skill discovery

El agent solo ve los skills que están en su skills_slugs[]. No puede llamar skills ajenos — defense-in-depth.

Persistencia

Cada call queda en agent_runs con tokens/cost; cada step en agent_run_steps; logs en S3 si exceden umbral.

Recovery

Heartbeats periódicos; si el worker muere, ClaimRun rescata el run vía FOR UPDATE SKIP LOCKED y resume desde el cursor.

7. Crons — disparo programado

internal/scheduler/cron/scheduler.go corre un loop de polling cada 30s. La idea clave: idempotencia multi-worker vía FOR UPDATE SKIP LOCKED, overlap protection vía INSERT condicional, y dispatch genérico a flow/agent/skill runner.

sequenceDiagram participant TK as ticker (30s) participant SCH as Scheduler participant SVC as cron.Service participant FR as flowrunner/agentrunner/skillrunner participant DB as Postgres loop cada 30s TK->>SCH: tick SCH->>SVC: PickDue(limit=50) SVC->>DB: SELECT FOR UPDATE SKIP LOCKED
WHERE next_run_at <= NOW() DB-->>SVC: due_crons[]
+ avanza next_run_at SVC-->>SCH: due_crons par per cron (goroutine) SCH->>SVC: StartExecution(cron_id)
INSERT...WHERE NOT EXISTS running alt overlap detectado SVC-->>SCH: skipped_overlap Note over SCH: no dispatch, history queda else OK SCH->>FR: dispatchSync(target_type, target_id, inputs) FR-->>SCH: result/error SCH->>SVC: FinishExecution(completed/failed) end end end

RLS FORCE (migration 000085): observations y sessions tienen RLS forzado. Sin el wireup SET LOCAL app.current_org_id, Postgres rechaza con "violates row-level security policy". Esto aplica a tools MCP, middleware HTTP, flow runner, agent runner. El bug clásico de hoy fue el flow runner sin tx — ya arreglado en commit 5244e58.
